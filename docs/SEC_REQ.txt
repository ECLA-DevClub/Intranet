MAIN REQUIREMENT
1. Цель и область применения

Настоящий регламент определяет:

правила обеспечения безопасности системы

требования к архитектуре, разработке и эксплуатации

порядок реагирования на инциденты

Применяется ко всем:

разработчикам

тестировщикам

менеджерам

администраторам инфраструктуры

2. Архитектура безопасности системы
2.1 Общая модель

Система должна быть построена по принципу:

Frontend (Next.js)

Backend API (Django + Wagtail)

База данных (PostgreSQL)

SSO (Google / Microsoft)

Reverse proxy (Nginx)

2.2 Принципы

Zero Trust — не доверять ни одному компоненту

Defense in Depth — многоуровневая защита

Least Privilege — минимальные доступы

Separation of Concerns — разделение компонентов

3. Управление доступом (Access Control)
3.1 Аутентификация

Только через SSO

MFA — обязательно для администраторов

Сессии:

ограниченное время жизни

HttpOnly cookies

3.2 Авторизация (RBAC)

Роли:

User

Content Editor

Admin

Требования:

Проверка прав на каждом API-запросе

Запрет доступа к чужим данным

4. Безопасность данных
4.1 Передача данных

Только через HTTPS (TLS 1.2+)

Запрет HTTP

4.2 Хранение данных

Пароли не хранятся (SSO)

Чувствительные данные:

шифруются (AES-256)

База данных:

доступ только из backend

4.3 Резервное копирование

Ежедневные backup’ы

Хранение:

отдельно от основной системы

Проверка восстановления — минимум раз в месяц

5. Безопасность разработки (Secure SDLC)
5.1 Этапы

Планирование

Разработка

Тестирование

Деплой

Поддержка

5.2 Требования к коду

Защита от:

XSS

CSRF

SQL Injection

Валидация всех входных данных

Запрет:

hardcoded secrets

5.3 Code Review

Каждый PR:

минимум 1 ревьюер

обязательная проверка безопасности

5.4 Зависимости

Регулярный аудит:

npm

pip

Уязвимости:

фиксируются до релиза

6. Безопасность API

Все endpoints:

требуют авторизацию

Ограничения:

rate limiting

Логи:

фиксируют доступ

7. Безопасность фронтенда

CSP (Content Security Policy)

Запрет inline scripts

Токены:

только HttpOnly cookies

Защита от:

XSS

clickjacking

8. Работа с файлами

Ограничение размера (например 50MB)

Проверка:

типа файла

Запрет:

выполнения файлов

Сканирование:

антивирус

9. Инфраструктурная безопасность
9.1 Серверы

Минимальный набор сервисов

Регулярные обновления

9.2 Сеть

Firewall

Закрытые порты

Доступ:

только по SSH (ключи)

9.3 CI/CD

Pipeline должен включать:

SAST

Dependency scanning

тесты

Запрещено:

деплой без проверок

10. Логирование и мониторинг

Логировать:

логины

ошибки

подозрительные действия

Запрещено:

логировать пароли и токены

11. Управление инцидентами
11.1 Классификация

Critical

High

Medium

Low

11.2 Процесс реагирования

Обнаружение

Изоляция

Анализ

Устранение

Отчёт

11.3 SLA

Critical: до 1 часа

High: до 24 часов

12. Тестирование безопасности

Регулярные:

penetration tests

automated scans

Проверка:

OWASP Top 10

13. Управление доступами

Регулярный аудит (раз в месяц)

Удаление:

неактивных пользователей

Контроль:

админ-доступов

14. Соответствие требованиям (Compliance)

Система должна соответствовать:

OWASP Top 10

GDPR (если есть персональные данные)

внутренним политикам компании

15. Обучение и осведомлённость

Обязательное обучение:

разработчиков

QA

Темы:

phishing

secure coding

16. Контроль и аудит

Регулярные проверки:

кода

инфраструктуры

Внешние аудиты (при необходимости)
