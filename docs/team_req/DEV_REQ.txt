Регламент кибербезопасности для разработчиков 
1. Общие принципы

Все разработчики обязаны соблюдать Secure SDLC (Secure Software Development Lifecycle).

Любой код считается небезопасным до прохождения:

Code Review

Static Analysis

Dependency Scan

Запрещено разрабатывать функционал без учёта OWASP Top 10.

2. Управление доступом и аутентификация

Использовать только корпоративное SSO (Google Workspace / Microsoft Entra ID).

Запрещено:

хранить пароли в коде

использовать hardcoded credentials

Все секреты:

хранятся в .env или vault (например HashiCorp Vault)

Принцип Least Privilege:

доступы выдаются минимально необходимыми

3. Безопасность кода (Backend — Django/Wagtail)
Обязательно:

Включить:

CSRF protection (по умолчанию в Django)

XSS protection

SQL injection protection через ORM

Запрещено:

raw SQL без параметризации

Валидация:

все входные данные валидируются (forms/serializers)

Работа с API:

Использовать:

JWT или session-based auth

Ограничить:

rate limiting

Все endpoints:

проверка прав доступа (RBAC)

4. Безопасность фронтенда (Next.js)

Запрещено:

хранить токены в localStorage (использовать HttpOnly cookies)

Обязательно:

Content Security Policy (CSP)

защита от XSS

Проверка входных данных:

на клиенте + обязательно на сервере

5. Работа с зависимостями

Обязательно:

pip-audit, npm audit

Запрещено:

использовать неподдерживаемые библиотеки

Регулярно:

обновление зависимостей

6. Работа с файлами и медиа (важно для CMS)

Ограничения:

max размер (например 50MB — как в документе)

Проверка:

MIME-type

antivirus scan

Запрещено:

выполнение загруженных файлов

7. Логирование и мониторинг

Логировать:

авторизацию

ошибки

подозрительные действия

Запрещено:

логировать пароли/токены

8. Шифрование

Обязательно:

TLS 1.2+

Sensitive data:

хранить в зашифрованном виде

9. Code Review

Каждый PR должен проверяться на:

безопасность

утечки данных

правильность RBAC

10. DevSecOps

CI/CD должен включать:

SAST (например SonarQube)

Dependency scanning

Запрещён деплой без проверок
